PoC 01 — Viabilidade da arquitetura ponta a ponta
2026-06-15 · 3 camadas executadas ao vivo com dados governamentais reais (IBGE). Documento: docs/pocs/01-viabilidade-e2e/poc-viabilidade-e2e.md

Objetivo

Verificar se a arquitetura de três camadas — ingestão config-driven, transformação dbt e integração semântica via LLM — é viável na prática, executando o processo completo de ponta a ponta com dados governamentais reais. A questão central: a promessa "adicionar uma fonte nova = apenas um arquivo YAML, sem código" se sustenta ao longo de todo o pipeline?

Método

Em vez de validar a arquitetura no papel, optou-se por executá-la de fato: subir a stack real (Apache Airflow + MinIO + DuckDB + PostgreSQL + dbt + Decision Layer) numa máquina local e fazê-la ingerir, transformar e integrar fontes públicas reais do IBGE. Cada etapa foi atravessada manualmente, registrando-se tanto o que funcionou quanto onde a arquitetura apresentou lacunas. Toda afirmação de resultado é embasada em logs reais capturados do sistema vivo (ver seção "Artefatos"), não em descrição redigida a posteriori. A escolha do IBGE como fonte deve-se a serem APIs públicas, estáveis e sem autenticação, com duas bases relacionáveis por uma chave comum (município → estado) — adequadas para exercitar também a camada de integração.

Como foi feito — execução por camada

Camada 1 (ingestão). Foram criados três arquivos YAML novos (IBGE municípios, IBGE estados e BCB PTAX) declarando cada fonte, sem escrever nenhuma linha de código de DAG. O Airflow gerou automaticamente um DAG ingest_api_* por YAML a partir de um único arquivo api_dag.py. O DAG do IBGE municípios foi então disparado e executado de verdade: a API foi consultada, o artefato bruto gravado em MinIO (camada bronze) e a tabela limpa materializada em DuckDB (camada silver).

Camada 2 (transformação). Para o dado chegar ao dbt foi necessário transpô-lo do DuckDB para um PostgreSQL analítico (ponte inexistente no projeto) e declarar a fonte e o modelo bronze manualmente, em SQL — pois o dbt é acoplado às fontes pré-existentes. O dbt run então materializou a tabela bronze do IBGE.

Camada 3 (integração semântica). Duas tabelas IBGE que atravessaram o pipeline (municípios e estados) foram submetidas à Decision Layer, que usa raciocínio LLM (executado via assinatura Claude Code) para identificar a chave de junção entre bases. O resultado foi comparado com os candidatos gerados pela análise estatística automática.

Resultado

A arquitetura é viável. A camada de ingestão cumpriu integralmente a promessa config-driven: três fontes novas foram adicionadas só com YAML, e o dado real do IBGE (5.571 municípios) atravessou API → MinIO → DuckDB sem código por fonte. Na camada de integração, o raciocínio semântico produziu a evidência mais forte da tese: identificou a chave correta uf_id ↔ id mesmo com match_rate estatístico de 0% — diagnosticando que se tratava de um falso negativo por incompatibilidade de tipo (float vs. inteiro) — e rejeitou um candidato concorrente de score numérico maior por reconhecê-lo como chave derivada redundante. O LLM acertou onde a heurística estatística falhou completamente.

A PoC também delimitou com precisão onde a arquitetura ainda não se sustenta: a promessa config-driven não se estende à transformação (o dbt exige SQL manual por fonte — a "Costura B", lacuna central), e as camadas não compartilham storage (DuckDB, PostgreSQL e CSV não se conectam sem pontes ausentes no código). Por fim, quatro bugs que impediam a execução ponta a ponta foram encontrados e corrigidos, evidenciando que a arquitetura havia sido projetada mas nunca executada de fato. As lacunas são trabalho futuro bem delimitado, não falhas conceituais — exatamente a fronteira que um trabalho de conclusão deve mapear com honestidade.
