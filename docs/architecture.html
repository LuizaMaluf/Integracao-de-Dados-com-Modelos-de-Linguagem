1 Visão Geral
Sistema completo

Da fonte governamental à Integration Key identificada

O Gov Hub recebe dados de sistemas governamentais heterogêneos, ingere via pipeline Medallion e usa um agente semântico para identificar automaticamente como conectar tabelas de sistemas distintos — sem intervenção manual. Tudo dentro do repositório gov-hub, em dois módulos: ingestion/ e integration/.

2 Módulo ingestion/
Ingestão

Componentes do pipeline de ingestão

# Componente Responsabilidade Ferramenta Onde vive
1 Source Registry Declaração de cada fonte: URL, auth, paginação, schedule, schema alvo, conflict fields YAML + Pydantic ingestion/configs/
2 DAG Factory Lê o Source Registry e gera todos os DAGs do Airflow automaticamente — sem arquivo Python por fonte astronomer/dag-factory ingestion/dag_factory.yaml
3 Generic Extractor HTTP com retry e backoff, paginação parametrizada (cursor/page/offset), auth por config, buffer IO.Bytes httpx ingestion/extractors/base.py
3b Extractors especializados PDF (layout analysis + Claude API), CSV/XLSX, SQL dump — lógica que não cabe no extractor genérico pdfplumber · unstructured · psycopg2 ingestion/extractors/
4 Bronze Store Artefato bruto imutável por execução. Path: raw/<source>/<date>/ MinIO (S3-compatible) ingestion/storage/bronze.py
5 Parsers Converte artefato bronze para DataFrame estruturado (semântico via LLM ou estrutural via regras) pandas · Claude API ingestion/parsers/
6 Transform + Validate Normaliza colunas, valida schema Pandera declarado no YAML da fonte, rejeita inválidos pandera ingestion/storage/silver.py
7 Silver Store Tabelas estruturadas por schema de origem, acessíveis por múltiplos processos PostgreSQL ingestion/storage/silver.py
8 Table Catalog Registra cada carga: fonte, tabela, schema, linhas, quality_ok, timestamp catalog.ingested_tables (PostgreSQL) ingestion/storage/catalog.py
9 Integration Bridge Consulta o Catalog, carrega os DataFrames do silver e aciona o IntegrationAgent Python ingestion/bridge/integration_bridge.py

3 Módulo integration/
Agente semântico

IntegrationAgent — identificação automática de chaves de integração

Dado dois DataFrames (tabela A e tabela B), o agente identifica qual campo de A conecta com qual campo de B. O pipeline passa por cinco estágios: perfilagem estatística → geração de candidatos → camadas de evidência (semântica e de conteúdo) → decisão com LLM → relatório.

Estágio Componente Output Detalhe técnico
E1 TableProfiler TableMetadata dtype, cardinalidade, nulos, unicidade, top valores, comprimento médio, padrão regex — por coluna
E2 CandidateGenerator CandidateKey[] Score semântico ≥ 0.30 → caminho semântico. Abaixo → Content Evidence Layer como fallback. Score composto: semântico×0.35 + match_rate×0.30 + structural×0.20 + pattern×0.15
E3a Semantic Evidence Layer SemanticScore Embeddings de nomes de coluna via sentence-transformers. Threshold 0.30 para promoção direta ao Decision Layer.
E3b Content Evidence Layer ContentEvidence format_match · overlap_rate · substring_match_rate · content_score. Pares com content_score ≥ 0.50 são promovidos mesmo com score semântico baixo. Detecta Derived Key (chave SIAFI embutida como substring).
E4 Decision Layer IntegrationKey + confiança Claude API raciocina sobre os candidatos ranqueados e escolhe a melhor Integration Key. Busca Derived Key para identificadores SIAFI (NC, NE, PF) embutidos em campos compostos.
E5 ReportGenerator HTML + JSON Gauge de confiança, score bars por candidato, evidências, código Python e SQL prontos para o join.

3 Módulo transformation/
dbt · independente do Airflow

Transformação — separada da orquestração

O módulo transformation/ contém o projeto dbt do gov-hub. Ele é completamente independente do Airflow: pode ser executado via CLI (dbt run) sem nenhum container em pé. O Airflow apenas agenda e monitora a execução via DAG Cosmos — mas não possui os arquivos dbt dentro de si.

Separação de responsabilidades:
  • ingestion/ — extrai e carrega dados brutos no silver (PostgreSQL). Responsabilidade: mover dados.
  • transformation/ — transforma o silver em modelos analíticos. Responsabilidade: modelar dados.
  • O Airflow aciona o dbt via transformation_dag.py, mas os arquivos dbt estão fora do diretório de DAGs.

Estrutura

transformation/ ├── dbt_project.yml ← projeto gov_hub ├── profiles.yml ← conexão PostgreSQL ├── packages.yml ← dbt-utils ├── .env.example └── models/ ├── bronze/ ← leitura do silver (incremental) │ ├── sources.yml ← declara tabelas do silver │ └── siafi_notas_empenho.sql ├── silver/ ← normalização e enriquecimento │ └── siafi_empenhos_enriched.sql └── gold/ ← modelos para Superset └── painel_execucao_orcamentaria.sql

Como rodar

Fora do container (dev):

cd gov-hub/
dbt run --project-dir transformation/ \
        --profiles-dir transformation/
dbt test --project-dir transformation/ \
         --profiles-dir transformation/

Via Airflow (produção):

A DAG gov_hub_transformation roda diariamente às 01:00 via Cosmos, apontando para /opt/airflow/transformation (volume montado separado de /opt/airflow/dags).

5 Integration Gateway — ponte entre os módulos
Bridge

Como o ingestion/ aciona o integration/

O Integration Gateway conecta os dois módulos do repositório. Ele vive em ingestion/bridge/ e é acionado por uma DAG Airflow dedicada. Consulta o Table Catalog para descobrir tabelas disponíveis, carrega os DataFrames do silver e delega ao IntegrationAgent a identificação da chave.

Table Catalog — catalog.ingested_tables filtra quality_ok = true · seleciona par de tabelas para integrar
↓
carrega os dois DataFrames do silver
Integration Gateway — ingestion/bridge/integration_bridge.py DAG: integration_bridge_dag · recebe table_a e table_b como parâmetros · constrói TableMetadata
↓
chama IntegrationAgent.run(df_a, meta_a, df_b, meta_b)
IntegrationAgent — integration/src/agent/orchestrator.py E1 Perfilagem → E2 Candidatos → E3 Evidências → E4 Decision Layer LLM
↓
integration/output/identificar_chave_<a>__<b>.json Integration Key recomendada · score de confiança · código Python e SQL prontos para o join
Exemplo: após ingerir siafi.notas_empenho e transfere_gov.planos_acao, o gateway aciona o agente que identifica num_empenho ↔ nr_empenho como Integration Key, com código Python e SQL gerado automaticamente para o join entre os dois sistemas.
Acionamento da DAG: 
{"table_a": "siafi.notas_empenho", "table_b": "transfere_gov.planos_acao"}
Pode ser disparado manualmente via Airflow UI ou automaticamente por um sensor que monitora o Catalog em busca de pares novos.

7 Decisões de Arquitetura (ADRs)
ADR 0001

Caminho único de notificação via on_failure_callback

Todo alerta passa pelo on_failure_callback do Airflow. A task validate_volume levanta exceção com texto descritivo em vez de notificar diretamente, evitando mensagens duplicadas.

Aceito
ADR 0002

Substituir DAGs individuais por DAG Factory config-driven

Um dag_factory.yaml lido pelo astronomer/dag-factory substitui todos os DAGs hard-coded. Adicionar uma fonte governamental nova = criar um YAML, sem arquivo Python.

Proposto
ADR 0003

Bronze Store (MinIO) antes do Silver

Cada ingestão escreve o payload bruto em MinIO antes de qualquer transformação. Permite reprocessar sem re-hit na API. A mudança de licença de Mai/2025 removeu o web console do MinIO Community Edition, mas não afeta a API S3 usada pelo código.

Proposto
ADR 0004

Integration Gateway como ponte entre ingestion/ e integration/

O gateway vive em ingestion/bridge/, consulta o Table Catalog e aciona o IntegrationAgent via DAG separada — desacoplando ingestão de identificação de chaves.
