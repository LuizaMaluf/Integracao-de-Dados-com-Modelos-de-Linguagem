O Problema

O governo federal brasileiro opera dezenas de sistemas estruturantes — SIAFI, TransfereGov, ComprasGov, SIAPE, SICONV, SIORG, entre outros — cada um com modelo de dados próprio, vocabulário diferente e sem mecanismo oficial de integração entre si. Gestores públicos que precisam cruzar dados de execução orçamentária com contratos, ou empenhos com planos de ação, fazem isso manualmente ou não fazem.

Situação encontrada

  • 73 arquivos de DAG Airflow — um por endpoint de cada sistema
  • 15+ classes ClienteXXX com lógica de HTTP duplicada
  • Dados vão direto da API para o PostgreSQL — sem preservação de artefato bruto
  • Schema criado dinamicamente em runtime (create_table_if_not_exists) — drift silencioso
  • dbt acoplado dentro da pasta de DAGs do Airflow
  • Identificação de chave de join entre sistemas: 100% manual
  • Adicionar nova fonte governamental: criar 5+ arquivos Python

O que queremos

  • Adicionar uma fonte = criar 1 arquivo YAML
  • Dado bruto preservado para reprocessamento sem re-puxar da API
  • Schema gerenciado por versionamento, não por código runtime
  • dbt independente do Airflow — roda via CLI sem container
  • Identificação automática de chave de join com evidência semântica + LLM
  • Catálogo de dados rastreável e consultável
  • Pipeline auditável ponta a ponta
Impacto real: sem integração automática entre sistemas, um relatório que cruza empenhos do SIAFI com planos de ação do TransfereGov requer análise manual de schemas de dois sistemas distintos, construção de query ad hoc e validação por especialista. Com o Gov Hub BR, esse processo é automatizado pelo IntegrationAgent — que identifica a chave, calcula a confiança e gera o código Python e SQL pronto.

Processo de Criação — Linha do Tempo

1
Diagnóstico inicial

Análise do pipeline existente

Mapeamento do data-application-gov-hub: 73 DAGs individuais, 15+ classes HTTP quase idênticas, dados indo direto de API para PostgreSQL sem bronze layer, dbt acoplado dentro da pasta de DAGs do Airflow. Identificamos os quatro gargalos principais: escalabilidade, preservação de dados, acoplamento dbt/Airflow e identificação manual de chaves de integração.

2
Decisão de arquitetura

Adoção do padrão Medallion com pipeline config-driven

Escolhemos a Medallion Architecture (Bronze → Silver → Gold) como padrão estrutural por ser o padrão de mercado para data lakehouses (adotado por Databricks, Microsoft Fabric, Snowflake). A abordagem config-driven elimina o código duplicado: cada fonte governamental é declarada em um YAML, e o dag-factory da Astronomer gera os DAGs automaticamente.

3
Bronze layer

MinIO como object store imutável

Introduzimos o MinIO (S3-compatible, self-hosted, open source) como Bronze Store. Cada execução de ingestão grava o payload bruto da API em raw/<source>/<date>/response.json antes de qualquer transformação. APIs governamentais como o Portal da Transparência têm histórico de mudanças retroativas — o bronze preserva a fidelidade original do dado.

4
Separação de responsabilidades

transformation/ como módulo independente

O dbt foi movido para transformation/, fora do diretório de DAGs. O Airflow agenda a execução via DAG Cosmos, mas não possui os arquivos dbt. Isso permite: (1) rodar dbt run via CLI sem container Airflow em pé; (2) versionar e revisar modelos dbt independentemente dos DAGs de ingestão; (3) times diferentes trabalharem nos dois módulos sem conflito.

5
Inovação central do TCC

IntegrationAgent — identificação automática de chaves de integração

Desenvolvemos o componente mais inovador do projeto: um agente que recebe dois DataFrames e identifica automaticamente qual campo de um conecta com qual campo do outro. O pipeline usa embeddings semânticos (sentence-transformers), análise de conteúdo (sobreposição de valores, padrão regex, substring matching) e raciocínio LLM (Claude API) para decidir a Integration Key com score de confiança e código de join gerado automaticamente.

6
Integração dos módulos

Integration Gateway — ponte entre ingestão e agente

O Table Catalog registra cada carga bem-sucedida no silver. O Integration Gateway consulta o Catalog, carrega os DataFrames e aciona o IntegrationAgent. O resultado — Integration Key recomendada, score, código Python e SQL — alimenta modelos Gold no dbt, fechando o ciclo: dado bruto → silver → chave identificada → modelo analítico → Superset.

4

Decisões Técnicas e Justificativas

ADR 0002

Source Registry YAML + DAG Factory

Escalabilidade
Problema

73 arquivos Python de DAG com estrutura quase idêntica. Adicionar uma fonte = criar novos arquivos, duplicar lógica, risco de inconsistência.

Decisão

Um YAML por fonte no Source Registry. O astronomer/dag-factory lê os YAMLs e gera os DAGs automaticamente. Nenhum Python por fonte.

Consequência

Adicionar uma nova fonte governamental leva ~5 minutos (escrever o YAML). Mudança transversal (ex: retry delay) = editar 1 lugar.

ADR 0003

Bronze Store — MinIO antes do Silver

Auditabilidade
Problema

API → PostgreSQL direto. Sem artefato bruto: se a API mudar, dado histórico está perdido. Impossível auditar o que foi ingerido originalmente.

Decisão

MinIO (S3-compatible, self-hosted) recebe cada payload bruto antes de qualquer transformação. Uma escrita por execução, imutável.

Consequência

Reprocessamento sem nova chamada à API. Auditoria completa do dado na fonte. Conforme padrões de dados abertos governamentais.

Novo

transformation/ separado do Airflow

Separação de responsabilidades
Problema

dbt dentro de dags/dbt/ — acoplado ao Airflow. Para testar um modelo dbt, era preciso o container Airflow em pé. Times de ingestão e transformação conflitavam no mesmo diretório.

Decisão

transformation/ na raiz do repo, independente. Montado via Docker volume separado. DAG Cosmos aponta para o path externo.

Consequência

dbt run --project-dir transformation/ funciona sem Airflow. Branches de modelos dbt não afetam DAGs de ingestão. CI pode testar dbt e Airflow separadamente.

ADR 0004

IntegrationAgent com Evidence Layer + LLM

Contribuição de pesquisa
Problema

Identificar qual campo do SIAFI conecta com qual campo do TransfereGov é um processo 100% manual, que requer especialista de domínio e horas de análise para cada par de sistemas.

Decisão

Pipeline em 4 estágios: perfilagem estatística → candidatos por score semântico (embeddings) → Evidence Layer (conteúdo + estrutura) → Decision Layer (LLM reasoning).

Consequência

Integração entre dois sistemas governamentais identificada em segundos com score de confiança e código de join gerado. Processo antes de horas → agora automatizado.

ADR 0005

Context Store Passivo — quatro tabelas no schema context

Arquitetura Transversal
Problema

O Integration Gateway recalcula tudo do zero a cada run. Os grupos de domínio ficam hardcoded em Python (DOMAIN_SEMANTIC_GROUPS) — novos domínios exigem edição de código. Contextos customizados criados via /definir-contexto não sobrevivem entre runs.

Decisão

Quatro tabelas no schema context: domain_contexts (vocabulários persistidos, substitui o dict Python), column_annotations, table_profiles e integration_discoveries (com justificativa LLM completa e flag is_validated). Schema gerenciado pelo dbt — não por código runtime.

Consequência

Novos domínios entram via INSERT em domain_contexts — sem PR de código. O mapear-integracoes parte das anotações já acumuladas. Conhecimento de integração persiste e melhora a cada run. Override humano via is_validated = FALSE invalida descobertas incorretas.

ADR 0006

Profile em dois momentos + ContextResolver como mediador dbt

Performance + Desacoplamento
Problema

Profile completo durante a ingestão bloqueia o stage_silver. O dbt não consegue consultar integration_discoveries diretamente para decidir quais modelos executar — não há mediador entre o Context Store e o cosmos.

Decisão

DAG revisada: extract → write_bronze → stage_silver → annotate_context (leve) e ↘ profile_table (pesado, paralelo). ContextResolver consulta o store antes do DbtTaskGroup e retorna o --select correto: só models/bronze/ sem integração confirmada, models/ completo com.

Consequência

Falha do profile_table não bloqueia a ingestão. O dbt executa apenas os modelos que fazem sentido dado o estado atual das integrações — sem modelos gold orfãos rodando sem joins definidos.

ADR 0007

Invalidação diferenciada + find_domain_group conectado ao banco

Consistência
Problema

Re-ingestão pode deixar annotations desatualizadas. Discoveries com confiança alta mas incorretas não têm mecanismo de correção. find_domain_group só conhece domínios compilados em Python.

Decisão

column_annotations e table_profiles: upsert simples — estado atual é a verdade. integration_discoveries: upsert com GREATEST(confidence) + is_validated para override humano. find_domain_group consulta context.domain_contexts antes do fallback Python.

Consequência

Annotations sempre refletem o schema atual. Discoveries acumulam sem crescimento descontrolado. Analistas podem rejeitar discoveries incorretas via is_validated = FALSE sem apagar o histórico. Grupos de domínio extensíveis sem editar código.

O IntegrationAgent — Contribuição Central

Evidence Layer + LLM Reasoning

Pipeline de identificação automática de chaves

O IntegrationAgent é a contribuição de pesquisa central do TCC. Dado dois DataFrames de sistemas governamentais distintos, o agente identifica automaticamente qual campo conecta as duas tabelas — sem intervenção humana. O pipeline combina três técnicas complementares.

Arquitetura Final — Três Módulos

ingestion/

Orquestração Airflow + extração das APIs governamentais + Bronze Store (MinIO) + Silver (PostgreSQL) + Table Catalog.

Airflow MinIO PostgreSQL httpx pandera
transformation/

Projeto dbt independente. Bronze → Silver → Gold. Modelos analíticos para o Superset. Roda via CLI sem Airflow.

dbt PostgreSQL Superset
integration/

IntegrationAgent. Identifica automaticamente chaves de join entre sistemas governamentais com NLP + LLM.

sentence-transformers Claude API pandas
Princípio arquitetural: cada módulo tem uma única responsabilidade e pode ser executado e testado de forma independente. O ingestion/ não conhece o dbt. O transformation/ não conhece o Airflow. O integration/ não depende de nenhum dos dois — recebe DataFrames e devolve uma Integration Key.

Benefícios

📈

Escalabilidade

Adicionar qualquer novo sistema governamental (SCDP, SIOP, SEI) = criar um YAML de ~20 linhas. Zero código Python novo. Zero risco de regressão em fontes existentes.

✅

Qualidade de dados

Pandera valida o schema declarado no YAML da fonte antes de qualquer dado chegar ao silver. Registros inválidos falham a task do Airflow — não silenciosamente corrompem o banco.

🤖

IA para integração

O IntegrationAgent identifica automaticamente chaves de join entre sistemas governamentais com NLP + análise de conteúdo + LLM. Processo antes manual de horas passa a ser automatizado.

🔍

Auditabilidade

Bronze Store preserva o dado bruto imutável. Table Catalog registra cada carga com timestamp, row count e status de qualidade. Rastreabilidade completa de onde cada dado veio e quando entrou.

🔀

Independência entre times

Engenheiros de ingestão trabalham em ingestion/. Analistas de dados trabalham em transformation/ sem container Airflow. Times paralelos sem conflito de arquivos.

🔓

100% open source

Airflow, MinIO, PostgreSQL, dbt, sentence-transformers, pandera. Nenhuma dependência de serviço gerenciado ou licença proprietária. Replicável por qualquer órgão público.

Decisões Arquiteturais Registradas

ADR 0001
Caminho único de notificação via on_failure_callback

Todo alerta Telegram passa por um único ponto — o callback do Airflow. A task de validação de volume levanta exceção descritiva em vez de notificar diretamente, evitando flood de mensagens duplicadas.

Aceito
ADR 0002
Source Registry YAML + DAG Factory

73 arquivos de DAG e 15+ clientes HTTP substituídos por YAML declarativos e dag-factory. Fonte nova = YAML de 20 linhas.

Proposto
ADR 0003
Bronze Store com MinIO antes do Silver

Artefato bruto preservado antes de qualquer transformação. Permite reprocessamento sem nova chamada à API. MinIO AGPL-3.0, a mudança de licença de Mai/2025 não afeta a API S3 usada no código.

Proposto
ADR 0004
Integration Gateway como ponte entre ingestion/ e integration/

O gateway consulta o Table Catalog para descobrir pares de tabelas e aciona o IntegrationAgent via DAG separada, desacoplando ingestão de identificação de chaves.

Proposto
ADR 0005
Context Store Passivo — quatro tabelas no schema context

domain_contexts (vocabulários de domínio persistidos, substitui o dict Python hardcoded), column_annotations (domain_group por coluna, escrita na ingestão), table_profiles (perfil semântico, escrita em task separada), integration_discoveries (chaves com justificativa completa do LLM e flag is_validated para override humano). O schema é criado e versionado pelo dbt — não por código runtime.

Proposto
ADR 0006
Profile em dois momentos + ContextResolver como mediador dbt

Na ingestão: task annotate_context anota apenas domain_group (custo zero). Task profile_table roda separada pós-stage_silver — se falhar, a ingestão já está salva. O ContextResolver consulta integration_discoveries antes do DbtTaskGroup e decide o --select adequado: só models/bronze/ se não há integração confirmada, models/ completo se há.

Proposto
ADR 0007
Invalidação diferenciada + find_domain_group conectado ao banco

column_annotations e table_profiles: upsert simples — estado atual é a verdade. integration_discoveries: upsert com GREATEST(confidence) preserva o melhor resultado; campo is_validated (NULL = automático, TRUE = confirmado, FALSE = rejeitado) permite override humano sem perda do histórico. find_domain_group consulta context.domain_contexts antes do fallback Python, tornando os grupos de domínio extensíveis sem editar código.
